bliterati (www.bliterati.com)

commentary from 2012

My primary coding example is bliterati. It's a publishing app hosted on Google App Engine. App Engine is Google's cloud platform and allows Python and Java based backends (well I guess there's also a Go backend). I use python and the webapp framework as well as many django libraries and the django template system. I've of course also written a lot of HTML, CSS, and Javascript for bliterati. All told the site uses several thousand lines of code.

It maintains a lot of user information. All pages are dynamically generated. It has a fairly sophisticatd Slashdot-like comment system. Users can share their drafts of documents with a self-selected group of users and those users can create revisions in which they've highlighted and otherwise marked up the original. Documents can be tagged, and can be replies to one another. It dynamically generates user specific HTML email updates. Pretty much all of these features are coded at a fairly low level. While I appreciate the advantages of using libraries and frameworks to accelerate the development process, I've felt that, at least during this learning phase, creating my own code for these things gives some valuable experience.

All the code is hosted in a public repository on GitHub. Virtually all of the interesting python code is in either main.py or messages.py. Main.py contains all of the request handlers for the site and messages.py contains the subroutines that generate email messages. I would say that messages.py is an example of reasonably good Python code.

Most of the javascript code can be found in the static/js folder. One of the cleaner pieces of javascript code would be tag_browser.js. It uses JSON and AJAX to power the tag browser that you see on the home page of bliterati.

If you do peruse the code some, you'll almost certainly find a lot that is less than ideal. I'd ask that you keep in mind that over the course of the year that I've been working on bliterati I've gone from being an absolute novice in web programming (though a somewhat skilled programmer) to demonstrating a moderate level of expertise. I've cobbled together the knowledge I need to make bliterati's bits and pieces work from books and the web. While I've rewritten many things in the code, it is still a patchwork that was written at various points along my learning curve. If you probe deeply enough you're likely to find:

  • Invalid HTML.
  • Javascipt that would give JSLint seizures.
  • Javascript that does not fail gracefully.
  • A lack of testing.
  • Design that is not compatible across all browsers (particularly legacy IEs).
  • Less than stellar software design.
  • Incomplete documentation.

Some of this is the product of my initial inexperience. A big part is that bliterati is an exercise in learning by doing, and I've deliberately prioritized maintaining momentum over worrying about writing perfect code. Bliterati is a pretty ambitious project for a solo, inexperienced, web programmer. So another source of these problems is a tendency to shirk tasks that would be more essential (or possible) when coding with others. My long-standing objective has been to get a prototype up and working and then to go back and refine and polish.

Anyhow, my point in going into all of that is to say that I hope you judge my code as just a step on the road to programming maturity. Hopefully it's a testament to my capacity to write serious code that works. I also hope that it demonstrates my capacity to make great strides working and learning independently. I would love to take my skills to the next level. I think that will happen more quickly if I get the chance to work closely with experienced professional programmers.
